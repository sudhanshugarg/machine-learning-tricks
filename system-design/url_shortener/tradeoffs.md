# URL Shortener - Design Tradeoffs

## Core Tradeoffs

### 1. Short Code Length vs Collision Risk

| Length | Possibilities | Space |
|--------|--------------|-------|
| 4      | 62^4 ≈ 1.5M  | Too small for scale |
| 5      | 62^5 ≈ 92M   | Still limited |
| 6      | 62^6 ≈ 56B   | **Good for 1-10B URLs** |
| 7      | 62^7 ≈ 3.5T  | Overkill for most use cases |

**Tradeoff:**
- **Shorter codes**: More user-friendly, cleaner URLs, but collision risk increases
- **Longer codes**: Safer, more combinations, but less elegant

**Decision:** Use 6 characters (56 billion combinations covers 1B+ URLs with buffer)

**Collision handling:**
- With 6 chars: ~0.001% collision risk for 1B URLs
- Use retry logic if collision detected
- Application-level retry is acceptable (rare occurrence)

---

### 2. URL Validation vs Storage

**Option A: Store all URLs**
- Pros: No validation overhead, faster writes
- Cons: Disk space, duplicate URLs, malicious content

**Option B: Validate before storing**
- Pros: Filter malicious URLs, detect duplicates, ensure viability
- Cons: Slower writes, external API calls (virustotal, phishing detection)

**Option C: Lazy validation**
- Pros: Fast writes, identify issues on redirection
- Cons: Bad user experience (broken links), security risk

**Decision:** Option A (store all) + Lazy validation
- Store immediately for speed
- Validate asynchronously in background
- Log problematic URLs for review

---

### 3. Custom Aliases vs Uniqueness

**Approach 1: Allow custom aliases**
- Pros: User-friendly, memorable, branded URLs
- Cons: Name collisions, higher storage, more complex logic

**Approach 2: Generate only**
- Pros: Simple, guaranteed unique, no conflicts
- Cons: Less user-friendly, harder to remember

**Approach 3: Hybrid**
- Pros: Best of both - generated by default, custom optional
- Cons: Need to handle both cases

**Decision:** Hybrid approach
- Generated codes: `http://short.io/abc123`
- Custom aliases: `http://short.io/mycompany` (premium feature)
- Prevent reserved words (admin, api, etc.)

**Database constraint:**
```sql
UNIQUE (user_id, custom_alias)  -- User-specific custom codes
```

---

### 4. Caching vs Storage

**Option A: Cache everything**
- Pros: Super fast, no DB queries
- Cons: Memory cost (50GB+), can't fit all URLs

**Option B: Cache hot data only (80/20)**
- Pros: 95% hit rate, manageable memory (50GB for 1M URLs)
- Cons: Cache misses require DB query

**Option C: No caching**
- Pros: Consistency, simplicity
- Cons: Every request hits DB, high latency (100ms+)

**Decision:** Option B (Cache hot data)
- Cache top 1% of URLs (get 95%+ hit rate)
- Use LRU eviction policy
- TTL: 24 hours
- Hit rate monitoring: Alert if <90%

**Cache strategy:**
```
Redis:
- 50 GB capacity
- 62^6 / 20 ≈ 2.8B URLs cacheable
- At 1-10B scale: ~0.28%-2.8% of URLs in cache
- But covers 90%+ of traffic
```

---

### 5. Database Sharding vs Simplicity

**Option A: Single database**
- Pros: Simple architecture, no sharding logic, easier to manage
- Cons: Limited scale (50M QPS max), single point of failure

**Option B: Read replicas only**
- Pros: Read scaling, simple writes
- Cons: Write bottleneck, replication lag

**Option C: Sharded database**
- Pros: Unlimited horizontal scale, high throughput
- Cons: Complex routing, transaction limitations, data migration

**Decision:** Start with read replicas, shard when needed
- Phase 1: Primary + 2-3 replicas (handles 1-10B URLs)
- Phase 2: Shard by short_code (10B+ URLs)

**Sharding key:**
```
shard_id = hash(short_code) % num_shards

Advantages:
- Equal distribution
- Cache locality
- Easy range queries
```

---

### 6. Analytics: Real-time vs Eventual Consistency

**Option A: Real-time counters**
- Pros: Accurate click counts immediately
- Cons: Heavy write load, DB bottleneck

**Option B: Async batch processing**
- Pros: Reduced DB load, simple logic
- Cons: 5-60 minute delay, inaccurate for real-time dashboard

**Option C: Dual write (cache + batch)**
- Pros: Fast metrics (from cache), accurate (from batch)
- Cons: Extra complexity, occasional inconsistency

**Decision:** Hybrid approach
- **Real-time metrics:** Cache counters (Redis)
- **Persistent metrics:** Batch write every 5 minutes
- **Accepted lag:** 5 minutes for analytics

**Implementation:**
```
Click event:
1. Increment Redis counter (1ms)
2. Publish to Kafka queue (async)

Every 5 minutes:
1. Read Redis counters
2. Batch write to time-series DB
3. Clear Redis counters
```

---

### 7. Expiration: Automatic vs Manual

**Option A: No expiration**
- Pros: Simple, links never break
- Cons: Unlimited storage growth

**Option B: Automatic expiration**
- Pros: Managed storage, privacy (links disappear)
- Cons: User surprised when links break

**Option C: Hybrid (optional expiration)**
- Pros: User choice, flexible
- Cons: More complex, need cleanup jobs

**Decision:** Option C with sane defaults
- Default: Never expires
- Optional: User can set expiration
- Soft delete: Mark as inactive, don't delete (data recovery)

**Cleanup strategy:**
```sql
-- Run nightly
DELETE FROM url_mappings
WHERE expires_at < NOW()
  AND soft_deleted = TRUE
  AND created_at < DATE_SUB(NOW(), INTERVAL 30 DAYS);
```

---

### 8. Location: Single vs Multi-region

**Option A: Single region**
- Pros: Simple, consistent data, easy management
- Cons: High latency for distant users, no failover

**Option B: Multi-region with replication**
- Pros: Low latency globally, failover capability
- Cons: Replication complexity, eventual consistency

**Option C: Multi-region with sharding**
- Pros: Low latency, high availability, scale
- Cons: Very complex, data migration hard

**Decision:** Start single-region, add regions as needed
- Phase 1: US-East (primary)
- Phase 2: Add US-West, Europe replicas
- Phase 3: Distributed shards if needed

**Implementation:**
```
Global Load Balancer
  ├─ US-East  (Primary)
  ├─ US-West  (Read-only replica)
  └─ Europe   (Read-only replica)

Writes: Always go to US-East (primary)
Reads: Route to nearest region (may be stale)
```

---

### 9. Error Handling: Fail-open vs Fail-closed

**Option A: Fail-open**
- Pros: Service stays available even with failures
- Cons: May serve stale/incorrect data

**Option B: Fail-closed**
- Pros: Data consistency, reliable
- Cons: Service unavailable during outage

**Decision:** Fail-open for redirects, Fail-closed for writes
- **Redirects:** If DB down, serve from cache (stale is OK)
- **Creating URLs:** If cache/DB down, return error (data integrity critical)
- **Analytics:** If analytics down, still count (degrade gracefully)

---

### 10. Authentication: Simple vs Robust

**Option A: API keys only**
- Pros: Simple, lightweight
- Cons: No session management, limited security

**Option B: OAuth + API keys**
- Pros: Third-party integration, secure
- Cons: Complex, external dependencies

**Option C: Basic auth + JWT**
- Pros: Good balance, industry standard
- Cons: Requires secure key management

**Decision:** API keys + JWT tokens
- Short-term: API keys for automation
- Long-term: JWT refresh tokens for sessions
- Store keys securely (hashed in DB)

## Latency vs Consistency Tradeoffs

### Read Operations (Redirects)

| Approach | Latency | Consistency | Cost |
|----------|---------|-------------|------|
| Cache only | 5ms | Eventual (24h) | Low |
| Cache + DB replica | 20ms | Strong | Medium |
| Cache + Primary DB | 50ms+ | Strong | High |
| **Chosen:** Cache-first | 10ms avg | Acceptable | Low |

**Justification:** Users tolerate stale content (24-hour delay acceptable)

---

### Write Operations (Creating URLs)

| Approach | Latency | Consistency | Cost |
|----------|---------|-------------|------|
| Cache only | 10ms | Weak | Low |
| Cache + async write | 50ms | Eventual (100ms) | Medium |
| **Chosen:** Sync write | 30ms | Strong | Medium |
| Quorum write | 100ms+ | Strong | High |

**Justification:** Users need confirmation immediately after creating URL

---

## Scalability vs Complexity

### Solution Approaches

| Scale | Approach | Complexity |
|-------|----------|-----------|
| 100K URLs | Single DB | Very low |
| 1M URLs | Primary + replicas | Low |
| 10M URLs | Sharded DB + cache | Medium |
| 100M+ URLs | Multi-region sharding | High |
| **Target:** 1-10B URLs | Replicas + cache | Medium |

## Cost vs Performance

### Infrastructure Investment

```
Budget: $10M/year

Scenario A: High performance
- 20 API servers: $2.4M
- Redis cluster (500GB): $1.2M
- Database (master + 3 replicas): $3.6M
- CDN bandwidth: $2M
- Monitoring, logs, misc: $0.8M
Total: $10M → P99 latency: 10ms

Scenario B: Balanced
- 10 API servers: $1.2M
- Redis cluster (100GB): $0.3M
- Database (master + 2 replicas): $2.4M
- CDN bandwidth: $0.8M
- Monitoring, logs, misc: $0.3M
Total: $5.0M → P99 latency: 50ms

Scenario C: Cost-optimized
- 5 API servers: $0.6M
- Redis (50GB): $0.15M
- Database (master + 1 replica): $1.2M
- CDN bandwidth: $0.3M
- Monitoring: $0.1M
Total: $2.35M → P99 latency: 200ms
```

**Decision:** Scenario B (Balanced)
- Acceptable latency for user experience
- Cost-effective
- Headroom for growth
- Can upgrade as traffic increases

## Key Decision Matrix

| Decision | Chosen | Reasoning |
|----------|--------|-----------|
| **Short code length** | 6 chars | Balance between URL length and collision risk |
| **Caching strategy** | Hot-only (1%) | 95%+ hit rate with manageable memory |
| **Database sharding** | Phase 2 | Start simple, shard when needed |
| **Analytics delay** | 5 minutes | Balance between accuracy and load |
| **Expiration** | Optional (default never) | User flexibility, simpler default |
| **Multi-region** | Phase 2 | Start single-region, add later |
| **Replication** | 2-3 replicas | Good HA, manageable complexity |
| **Error strategy** | Fail-open (reads) | Graceful degradation acceptable |
| **Authentication** | API keys + JWT | Balance security and simplicity |

---

## Summary

The URL shortener design prioritizes:
1. **Simplicity:** Start with straightforward architecture
2. **Scalability:** Clear path to 10B+ URLs
3. **Performance:** Sub-100ms latency for 99% of users
4. **Reliability:** Graceful degradation on failures
5. **Cost-effectiveness:** Balanced infrastructure investment

Trade-offs are intentional and well-documented for future evolution.
